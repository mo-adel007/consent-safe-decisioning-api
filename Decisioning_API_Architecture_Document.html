Decisioning API

Technical Architecture & Planning Document — Consent-Safe Personalisation
Date: 4 March 2026 Version: 1.0 Status: Pre-Implementation Review

1. Executive Summary

This document describes the architecture for a Decisioning API — a lightweight backend service that answers one question for every visitor:

"This visitor just opened this page. Which version of the content should they see?" The API evaluates rules and returns the correct variant, without the frontend needing to know the rules themselves.

Instead of rebuilding or redeploying a website for different audiences, the website asks the Decision API at runtime. This is the same architectural pattern used by experimentation platforms, marketing personalisation engines, feature flag systems, and growth experiment frameworks at scale.

2. The Big Picture — How Personalisation Works

2.1 The Runtime Flow

When a user opens a website, the following happens in real time:

1 User opens the website
↓
2 Frontend sends visitor context to the Decision API
↓
3 API receives the request at POST /decide
↓
4 API evaluates rules against visitor context
↓
5 API returns which variant to show (headline, flags, etc.)
↓
6 Frontend renders that variant to the user

3. Key Concepts

3.1 What Is a Variant?

A variant is simply a version of content. The same page can have multiple variants, each designed for a different audience.

Variant IDHeadlineTarget Audience
mobile-sale"Mobile users get 20% off"Mobile visitors from Egypt
seo-variant"Welcome Google visitors"Traffic from Google search
arabic-offer"مرحباً! اكتشف عروضنا"Arabic-speaking visitors
default"Welcome! Discover our offerings"Everyone else (fallback)

Each variant carries a headline (the text to display) and flags (boolean switches that control UI behaviour like showing banners or enabling chat). The frontend uses these to render the appropriate experience.

3.2 What Is the Headline?

The headline is the content payload associated with a variant. When the API says "show variant mobile-sale", the frontend receives:

  • headline: "Mobile users get 20% off" — the text to display prominently
  • flags: showDiscountBanner: true — tells the frontend to show a discount banner

The frontend uses these values to change the headline, show/hide banners, and enable/disable features — all without redeploying.

3.3 What Is the Rule Engine?

The rule engine is the condition matcher at the core of the API. It loops through a list of rules, checks if the visitor's context matches each rule's conditions, and returns the first matching variant.

How it works: For each rule → check if ALL conditions match → if yes, return that variant → if no rule matches, return the fallback.

Rules support AND logic — a rule can have multiple conditions, and all of them must be true for the rule to match. This allows for precise targeting like "visitors from Egypt AND on mobile devices".

Supported Operators

OperatorMeaningExample
eqExact equalitycountry eq "EG" → true if country is exactly "EG"
inValue is in a listcountry in ["EG", "SA", "AE"] → true if country is any of these
containsString contains substringreferrerDomain contains "google" → true for "google.com", "google.co.uk"

Key difference between the two flows: GET /config passes through the CDN and is cacheable — it rarely hits the origin server after the first request. POST /decide goes directly to the API because every visitor's context is unique and decisions cannot be cached.

4.3 The Two API Endpoints

Endpoint 1: GET /config/:siteId — "Give me the rules"

This endpoint returns the complete configuration for a given site: all the rules, all the variants, and the fallback. It is the rules database in API form.

Why does this endpoint exist separately?

  • Transparency — frontends, dashboards, or debugging tools can inspect the active ruleset
  • Cacheability — because config changes infrequently, we cache it aggressively at CDN and browser level
  • Separation of concerns — config retrieval is read-only; decision-making is a computation

Endpoint 2: POST /decide — "Decide for this visitor"

This is the core endpoint. The frontend sends everything it knows about the visitor (country, device, referrer, consent status, etc.) and the API responds with which variant they should see.

The /decide endpoint performs three steps internally:

  1. Load — fetch the ruleset for the given siteId from the config store
  2. Filter — apply the consent boundary (strip unsafe fields if marketing=false)
  3. Evaluate — run the rule engine top-to-bottom, return first match or fallback

5. Rules Configuration — Why Config Instead of Hardcoding

Marketing teams change targeting rules frequently. Today it might be "mobile users in Egypt get a discount", tomorrow it could be "visitors from Germany see a special campaign". Instead of redeploying the backend for every change, we separate the rules into a configuration layer.

The key insight: Config is data, not code. By treating rules as data that the engine reads, we can update targeting without any code deployment. This is how every production personalisation system works.

5.1 Example Ruleset for "site-nike"

PriorityRule IDConditions (AND)VariantHeadline
1 (highest) rule-eg-mobile country = "EG" AND deviceType = "mobile" mobile-sale "Mobile users get 20% off"
2 rule-google-seo referrerDomain = "google.com" seo-variant "Welcome Google visitors"
3 rule-arabic language = "ar" arabic-offer "مرحباً! اكتشف عروضنا"
Fallback — No conditions (always matches) default "Welcome! Discover our offerings"

Evaluation order matters. Rules are checked top-to-bottom. The first rule whose conditions ALL match wins. If no rule matches, the fallback is returned. This "first-match-wins" strategy is simple, predictable, and easy to reason about.

6. Consent Boundary — Privacy by Design

6.1 Why Consent Matters

Privacy regulations like GDPR require that users consent to marketing tracking. When a user sees a cookie consent popup and says "No" to marketing cookies, we are legally obligated to not use any data that could personally identify them.

6.2 The Consent Boundary Rule

When marketing = false: The system must NOT use any personal identifiers (visitorId, userId, email, traits, tracking IDs) to make decisions. Only anonymous, non-identifying signals may be used.

6.3 What Is Allowed vs. Blocked

FieldWhen marketing = trueWhen marketing = falseWhy
country ✓ Allowed ✓ Allowed Geographic region, does not identify a person
language ✓ Allowed ✓ Allowed Browser preference, does not identify a person
deviceType ✓ Allowed ✓ Allowed Device category (desktop/mobile/tablet), not identifying
referrerDomain ✓ Allowed ✓ Allowed Traffic source domain, not personal
visitorId ✓ Allowed ✗ BLOCKED Personally identifies a specific visitor
traits (plan, loginCount, etc.) ✓ Allowed ✗ BLOCKED Behavioural data tied to an individual
userId / email ✓ Allowed ✗ BLOCKED Direct personal identifier

6.4 How the Filter Works

┌──────────────────────────────────────────────────────────────────┐ │ INCOMING VISITOR CONTEXT │ │ │ │ visitorId: "vis-abc-123" │ │ country: "EG" │ │ language: "ar" │ │ deviceType: "mobile" │ │ referrerDomain: "google.com" │ │ traits: { plan: "premium", loginCount: 42 } │ └──────────────────────────────┬───────────────────────────────────┘ │ ┌─────────▼─────────┐ │ marketing = ? │ └─────────┬─────────┘ ┌─────────────┴─────────────┐ │ │ marketing = TRUE marketing = FALSE │ │ ▼ ▼ ┌──────────────────────────┐ ┌──────────────────────────┐ │ FULL CONTEXT PASSED │ │ CONSENT-SAFE ONLY │ │ │ │ │ │ visitorId ✓ │ │ country: "EG" ✓ │ │ country ✓ │ │ language: "ar" ✓ │ │ language ✓ │ │ deviceType: "mobile" ✓ │ │ deviceType ✓ │ │ referrerDomain: │ │ referrerDomain ✓ │ │ "google.com" ✓ │ │ traits ✓ │ │ │ │ │ │ visitorId ✗ DROPPED │ │ → Rule engine sees │ │ traits ✗ DROPPED │ │ everything │ │ │ └──────────────────────────┘ │ → Rule engine only │ │ sees safe fields │ └──────────────────────────┘

6.5 Practical Consequence

If two visitors have different visitorIds and traits, but the same country, language, deviceType, and referrerDomain — and marketing is false — they will always receive the exact same variant. This is the proof that personal data is not influencing the decision.

7. Caching Strategy — ETag & Cache-Control

7.1 Why Cache the Config?

The /config/:siteId endpoint returns rules that change infrequently (maybe a few times a day). Without caching, every page load triggers a request to the origin server. With caching, we dramatically reduce server load and improve response times.

7.2 The Caching Layers

Request Flow (left to right): ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌──────────┐ │ BROWSER │ ───▶ │ CDN │ ───▶ │ EDGE │ ───▶ │ ORIGIN │ │ │ │ │ │ │ │ SERVER │ │ Caches │ │ Caches │ │ Caches │ │ │ │ for 60s │ │ for 300s │ │ for 300s │ │ (truth) │ └──────────┘ └──────────┘ └──────────┘ └──────────┘ max-age=60 s-maxage=300 s-maxage=300 * Browser only asks CDN after 60 seconds * CDN only asks Origin after 300 seconds (5 minutes) * ETag enables 304 Not Modified (no body sent if unchanged)

7.3 Cache Headers Explained

HeaderValueWhat It Does
Cache-Control public, max-age=60, s-maxage=300 public — Any cache (browser, CDN, proxy) may store this response
max-age=60 — Browser considers response fresh for 60 seconds
s-maxage=300 — Shared caches (CDN/edge) consider response fresh for 300 seconds (5 minutes)
ETag "v3" (the configVersion) A fingerprint of the response content. When the cache expires, the client sends If-None-Match: "v3". If config hasn't changed, the server responds with 304 Not Modified (no body) — saving bandwidth.

7.4 How ETag Validation Works (Step by Step)

1 First request: Browser fetches GET /config/site-nike
↓
2 Server responds with full JSON body + header ETag: "v3"
↓
3 Browser caches the response for 60 seconds
↓
4 After 60 seconds, browser sends a new request with If-None-Match: "v3"
↓
5 Server checks: has config changed since v3?
↓
6a Unchanged: Server responds with 304 Not Modified (no body, very fast)
6b Changed: Server responds with new body + new ETag: "v4"
Result: Faster responses. Less bandwidth. Less server load. The CDN serves most requests without ever hitting the origin.

8. Data Flow — End-to-End Examples

8.1 Example: Egyptian Mobile User (marketing = true)

StepWhat HappensData
1. Frontend sends POST /decide siteId: "site-nike", country: "EG", deviceType: "mobile", marketing: true
2. Consent check marketing=true → full context All fields passed to rule engine (including visitorId, traits)
3. Rule evaluation Rule 1 checks: country="EG" AND deviceType="mobile" Both conditions match → ✓ MATCH
4. Response Return mobile-sale variant variantId: "mobile-sale", headline: "Mobile users get 20% off"

8.2 Example: Google Visitor from UK (marketing = false)

StepWhat HappensData
1. Frontend sends POST /decide siteId: "site-nike", country: "GB", referrerDomain: "google.com", visitorId: "vip-123", marketing: false
2. Consent check marketing=false → strip unsafe fields visitorId DROPPED. Only: country="GB", language, deviceType, referrerDomain="google.com"
3. Rule evaluation Rule 1: country="EG"? No (GB ≠ EG) → skip
Rule 2: referrerDomain="google.com"? Yes → ✓ MATCH		 First match wins at Rule 2
4. Response Return seo-variant variantId: "seo-variant", headline: "Welcome Google visitors"

Key point: Even though this visitor had a visitorId: "vip-123", it was dropped by the consent filter. The decision was made purely on the anonymous referrer domain signal.

8.3 Example: No Rules Match → Fallback

StepWhat HappensData
1. Frontend sends POST /decide siteId: "site-nike", country: "US", deviceType: "desktop", referrerDomain: "linkedin.com"
2. Rule evaluation Rule 1: country="EG"? No → skip
Rule 2: referrerDomain="google.com"? No → skip
Rule 3: language="ar"? No → skip		 No rules matched
3. Fallback Return default variant variantId: "default", headline: "Welcome! Discover our offerings"

13. Why Separate the Decision Engine from Content Rendering

An important architectural principle that engineers often miss: the Decision API only answers "what" to show, never "how" to show it.

ConcernResponsibilityWhere It Lives
WHAT variant to showDecision APIBackend
HOW to render itFrontend applicationClient

Benefits of This Separation

BenefitExplanation
Decoupled Deploys Rules can change without redeploying the frontend. Marketing adds a new rule? Just update config — the frontend adapts automatically.
Marketing Agility Marketing teams update rules in configuration, no engineering tickets needed for every campaign change.
Testability Decision logic is independently testable without rendering any UI. Pure input → output testing.
Multi-Channel The same Decision API can serve web, mobile app, email, push notifications — any channel that needs a decision.
Auditability All decisions flow through one service. Easy to log every decision for analysis, debugging, and compliance.
In one sentence: This project builds a small personalisation decision API that chooses which version of content a visitor should see, based on configurable rules and consent-safe data — keeping the decision logic completely separate from the rendering logic.
